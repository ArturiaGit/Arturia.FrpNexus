# FrpNexus UI/UX 设计简报 v1.0

## 1. 产品概述

FrpNexus 是一款面向个人开发者的开源 FRP 桌面部署与运维控制台。

它不是官网、营销页、云平台后台，也不是单纯的 FRP 配置编辑器。它的核心价值是把 FRP 的下载、部署、配置、启动、日志查看和基础诊断整合进一个桌面端，让用户不必频繁手写配置文件和 SSH 命令。

第一版产品重点支持：

- 通过 SSH/SFTP 管理远程 Linux 节点
- 自动下载或选择本地 FRP 核心文件
- 上传 `frpc` / `frps` 到远程节点
- 通过表单生成 TOML 配置
- 上传配置文件
- 启动、停止、重启远程 FRP 进程
- 查看日志、状态和基础错误信息

## 2. UI/UX 设计定位

FrpNexus 的界面应该像一个专业、清晰、可信赖的桌面运维工具。

产品是 Avalonia 桌面端应用，但视觉风格、控件气质和交互反馈应遵循 WinUI 3 / Fluent Design 的桌面应用体验。设计师可以参考 Windows 11 原生应用的布局、圆角、层级、命令栏、列表、表单和信息提示方式。

界面展示语言为中文。产品名称、FRP、SSH、SFTP、TOML、TCP、UDP、HTTP、HTTPS、Token、frpc、frps 等技术名词可以保留英文或原始大小写；导航、按钮、状态、表单标签、提示信息、空状态和错误说明应使用中文。

设计关键词：

- 专业
- 清晰
- 稳定
- 克制
- 本地优先
- 开发者友好
- 运维控制台感
- WinUI 3 风格
- 中文界面

不希望出现的方向：

- 营销官网式首页
- 大面积装饰性 Hero
- 过多渐变、插画、浮夸阴影
- 复杂企业后台感
- 游戏化或过度可爱风格
- 只像普通文本编辑器
- Web 后台模板感
- 英文 SaaS 仪表盘感

第一屏应该直接是可工作的产品界面，而不是欢迎页或产品介绍页。

## 3. 目标用户

核心用户是个人开发者、独立开发者、自托管爱好者和轻量运维用户。

典型用户特征：

- 拥有 VPS、云服务器、家用服务器或 NAS
- 需要把本地服务暴露到公网
- 可能使用 FRP 暴露 Web、API、SSH、RDP、NAS、游戏服务
- 能理解端口、域名、服务器、SSH 等基础概念
- 不希望频繁手写 FRP 配置和登录服务器操作
- 重视可控、透明、开源和本地化

设计时不要把用户当成完全不懂技术的新手，也不要把用户当成大型企业 SRE。界面应该降低操作成本，但保留足够的技术信息。

## 4. 核心使用场景

### 场景 A：部署一个新的 FRP 服务端

用户想在自己的 Linux VPS 上部署 `frps`。

流程：

1. 添加远程 Linux 节点
2. 填写 SSH 连接信息
3. 测试连接
4. 选择或下载 FRP 版本
5. 上传 `frps`
6. 创建 `frps.toml`
7. 上传配置
8. 启动 `frps`
9. 查看运行状态和日志

### 场景 B：创建一个本地服务穿透配置

用户想把本地 Web 服务或 SSH 服务暴露到远程服务器。

流程：

1. 选择已有服务端节点
2. 创建客户端配置 `frpc.toml`
3. 选择代理类型：TCP、UDP、HTTP 或 HTTPS
4. 填写本地地址、本地端口、远程端口或域名
5. 预览 TOML 配置
6. 保存配置
7. 上传配置或在本机运行
8. 查看隧道状态

### 场景 C：排查连接失败

用户发现隧道不可用。

流程：

1. 打开 Dashboard 或 Tunnels 页面查看异常状态
2. 进入对应节点或隧道详情
3. 查看最近错误日志
4. 检查远程进程是否运行
5. 检查端口监听状态
6. 重新上传配置或重启进程

## 5. 信息架构

建议采用桌面运维工具常见的信息架构：

```text
FrpNexus
├─ 仪表盘
├─ 节点
├─ 隧道
├─ 配置
├─ 运行
├─ 日志
└─ 设置
```

英文模块名可作为内部命名参考，但设计稿中主导航应使用中文。

### 仪表盘

作为产品第一屏，用于快速理解当前状态和进入常用操作。

需要展示：

- 节点总数
- 在线节点数
- 运行中的 FRP 进程数
- 活跃隧道数
- 最近错误
- 最近操作记录
- 快捷操作：添加节点、创建隧道、上传核心、查看日志

### 节点

管理远程节点。

需要展示：

- 节点名称
- 主机地址
- 系统类型
- 连接状态
- FRP 安装状态
- 最近连接时间
- 操作入口：测试连接、上传核心、打开日志、编辑

### 隧道

管理隧道配置和运行状态。

需要展示：

- 隧道名称
- 协议类型
- 所属节点
- 本地地址和端口
- 远程端口或域名
- 启用状态
- 运行状态
- 最近错误

### 配置

管理 `frps.toml` 和 `frpc.toml`。

需要支持：

- 表单编辑
- TOML 预览
- 配置校验
- 保存
- 上传
- 从远程拉取
- 导入和导出

### 运行

管理远程 FRP 运行状态。

需要支持：

- 启动
- 停止
- 重启
- 查询进程
- 查询端口
- 查看启动命令
- 后续预留 systemd 服务管理入口

### 日志

查看日志和错误。

需要支持：

- 节点筛选
- 进程筛选
- 日志搜索
- 错误高亮
- 复制日志
- 刷新
- 清空本地显示

### 设置

管理全局设置。

需要包含：

- FRP 下载源
- 默认远程安装目录
- SSH 密钥管理
- 本地数据存储位置
- 主题设置
- 日志显示偏好

## 6. 页面布局建议

推荐桌面端采用三段式结构：

```text
┌──────────────────────────────────────────────┐
│ Top Bar                                      │
├───────────────┬──────────────────────────────┤
│ Sidebar       │ Main Content                 │
│ Navigation    │                              │
└───────────────┴──────────────────────────────┘
```

### Top Bar

建议包含：

- 当前页面标题
- 全局连接状态
- 快捷操作按钮
- 设置入口
- 主题切换入口

### Sidebar

建议包含：

- 产品名称或图标
- 主导航
- 当前选中状态
- 底部版本信息或设置入口

### Main Content

根据页面展示列表、详情、表单、日志或状态面板。

建议使用列表加详情的结构，例如：

- 左侧为节点或隧道列表
- 右侧为详情、操作和日志摘要

## 7. 关键页面设计要求

### Dashboard 页面

Dashboard 不应该像营销首页，而应该是工作台。

建议模块：

- 状态概览区
- 快捷操作区
- 最近错误区
- 最近节点状态区
- 最近操作记录区

### Nodes 页面

Nodes 页面应支持快速扫描。

列表中应清楚显示：

- 在线 / 离线
- 已安装 / 未安装 FRP
- 正在运行 / 已停止
- 最近错误

节点详情中应包含：

- SSH 连接信息
- 远程安装目录
- FRP 版本
- 上传状态
- 运行状态
- 关联配置
- 最近日志

### Tunnel 编辑页面

隧道编辑应采用表单加预览结构。

推荐布局：

- 左侧或上方为配置表单
- 右侧或下方为 TOML 预览
- 底部为校验结果和操作按钮

表单字段应按代理类型动态变化。

TCP/UDP 需要：

- 隧道名称
- 本地 IP
- 本地端口
- 远程端口

HTTP/HTTPS 需要：

- 隧道名称
- 本地 IP
- 本地端口
- 自定义域名

### Logs 页面

日志页面应该信息密度较高，但不要混乱。

需要提供：

- 搜索框
- 节点筛选
- 日志级别筛选
- 自动刷新开关
- 错误高亮
- 复制按钮
- 跳到底部按钮

日志内容应使用等宽字体。

## 8. 交互规范

### 操作反馈

所有远程操作都必须有明确反馈。

包括：

- 连接中
- 上传中
- 启动中
- 停止中
- 重启中
- 成功
- 失败
- 超时

失败时不能只显示“失败”，应展示可理解的原因，例如：

- SSH 连接失败
- 认证失败
- 远程目录无权限
- FRP 文件不存在
- 配置校验失败
- 远程端口已被占用

### 危险操作确认

以下操作需要二次确认：

- 覆盖远程 FRP 核心
- 覆盖远程配置文件
- 停止正在运行的进程
- 重启正在运行的进程
- 启用或禁用 systemd 服务
- 删除节点
- 删除配置

确认弹窗应说明影响，不要只问“确定吗”。

### 表单校验

表单应即时校验：

- 必填项为空
- 端口格式错误
- 端口范围错误
- 节点未选择
- 域名格式错误
- 隧道名称重复
- Token 为空或过短

校验错误应出现在字段附近，同时在提交时汇总展示。

### 空状态

空状态要给出下一步操作。

示例：

- 没有节点时：引导添加第一个 Linux 节点
- 没有隧道时：引导创建 TCP 或 HTTP 隧道
- 没有日志时：提示启动 FRP 后会显示日志

## 9. 状态表达规范

建议设计统一状态系统。

节点状态：

- Online
- Offline
- Connecting
- Error
- Unknown

FRP 安装状态：

- Not Installed
- Installed
- Uploading
- Upgrade Available
- Failed

进程状态：

- Running
- Stopped
- Starting
- Stopping
- Restarting
- Failed

隧道状态：

- Enabled
- Disabled
- Active
- Inactive
- Error

日志级别：

- Info
- Warning
- Error
- Debug

状态颜色应克制且一致：

- 成功：绿色
- 警告：黄色或琥珀色
- 错误：红色
- 信息：蓝色
- 未知 / 禁用：灰色

## 10. 视觉风格建议

整体风格应偏专业工具，而不是品牌官网。视觉参考 WinUI 3 / Windows 11 桌面应用，强调清晰层级、轻量材质、克制圆角、明确焦点和稳定控件状态。

建议：

- 使用清晰的层级和留白
- 使用稳定的中性色背景
- 通过状态色表达信息
- 控件边界清楚
- 卡片只用于独立信息块，不要层层嵌套
- 日志、配置预览和命令内容使用等宽字体
- 表格和列表要便于扫描
- 使用 WinUI 3 风格的 NavigationView、CommandBar、InfoBar、TeachingTip、ContentDialog、ListView、DataGrid、ToggleSwitch、ProgressRing 等控件语义
- 使用 Windows 11 风格的轻量圆角、1px 分隔线、柔和层级和清晰悬停/选中状态

避免：

- 大面积紫蓝渐变
- 装饰性发光背景
- 过度圆角
- 过度阴影
- 低信息密度的大卡片堆叠
- 用营销文案占据主界面
- 过度拟物化或 macOS 风格
- Material Design 风格过强
- Web 管理后台模板化视觉

### 10.1 WinUI 3 / Fluent Design 规则

设计稿应遵循以下规则：

- 导航使用左侧 NavigationView 风格，图标加中文文字。
- 顶部操作区使用 CommandBar 风格，主操作靠右或与页面标题同区。
- 提醒信息使用 InfoBar 风格，按信息、成功、警告、错误区分状态。
- 确认弹窗使用 ContentDialog 风格，主按钮、取消按钮和危险按钮语义明确。
- 临时提示可使用 TeachingTip 或 Toast 风格，但不要遮挡主流程。
- 页面内容使用清晰分区，不使用多层卡片嵌套。
- 可点击列表项需要有 hover、selected、pressed、disabled 状态。
- 输入框、下拉框、按钮、开关、复选框应符合 Windows 桌面控件气质。
- 中文界面文案应短、直接、动作明确，例如“测试连接”“上传核心”“启动服务”“查看日志”。
- 错误提示应中文可读，技术名词保留原文，例如“SSH 认证失败”“远程端口已被占用”。

## 11. 设计资源与 Tokens

本节为 UI/UX 设计师提供可直接用于设计稿的基础资源。整体视觉应保持桌面工具感，避免过度品牌化和装饰化。

### 11.1 字体

中文与界面字体：

- Windows 首选：Segoe UI + Microsoft YaHei UI
- macOS 首选：PingFang SC
- Linux 首选：Noto Sans CJK SC
- 通用回退：system-ui, sans-serif

代码、日志、TOML 预览字体：

- 首选：JetBrains Mono
- Windows 备用：Cascadia Code, Consolas
- 通用回退：monospace

字号层级：

| 用途 | 字号 | 行高 | 字重 |
|---|---:|---:|---:|
| 页面标题 | 20 | 28 | 600 |
| 区块标题 | 16 | 24 | 600 |
| 表单分组标题 | 14 | 22 | 600 |
| 正文 / 表单标签 | 13 | 20 | 400 |
| 表格内容 | 13 | 20 | 400 |
| 辅助说明 | 12 | 18 | 400 |
| 状态标签 | 12 | 16 | 500 |
| 日志 / 代码 | 12 | 18 | 400 |

字体使用要求：

- 不使用负字距。
- 不随窗口宽度动态缩放字号。
- 日志、配置预览、命令和路径必须使用等宽字体。
- 按钮和状态标签文字应短，不依赖大字号表达层级。

### 11.5 间距、圆角与边框

间距基准采用 4px 网格。

| Token | 数值 | 用途 |
|---|---:|---|
| `space.1` | 4 | 紧凑内距、图标间距 |
| `space.2` | 8 | 控件内部间距 |
| `space.3` | 12 | 表单字段间距 |
| `space.4` | 16 | 面板内边距 |
| `space.5` | 20 | 区块间距 |
| `space.6` | 24 | 页面主要留白 |

圆角：

- 小控件：4px
- 输入框 / 按钮 / 状态标签：6px
- 面板 / 卡片 / 弹窗：8px
- 不使用超过 12px 的大圆角

边框：

- 默认边框：1px
- 焦点边框：1px 边框 + 2px focus ring
- 表格分隔线：1px
- 不使用厚重描边

### 11.6 阴影与层级

桌面端主界面应主要依赖边框和背景层级，不依赖强阴影。

| Token | 阴影 |
|---|---|
| `shadow.panel` | `0 1px 2px rgba(15, 23, 42, 0.08)` |
| `shadow.popover` | `0 8px 24px rgba(15, 23, 42, 0.16)` |
| `shadow.dialog` | `0 16px 40px rgba(15, 23, 42, 0.22)` |

使用规则：

- 常规面板优先使用边框，不默认加阴影。
- 弹窗、菜单、浮层可以使用阴影。
- 不使用发光、霓虹或大面积模糊阴影。

## 12. 尺寸与适配

FrpNexus 是桌面端应用，不是网页端产品。UI 需要适配桌面窗口缩放，但设计最大工作分辨率按 `1280 x 800` 控制。

桌面端尺寸要求：

- 最大设计分辨率：1280 x 800
- 推荐设计稿尺寸：1280 x 800
- 最小可用窗口：1100 x 720
- UI 需要在 1100 x 720 到 1280 x 800 之间自适应
- 不以 1440 x 900、1920 x 1080 或超宽屏作为主要设计基准

布局要求：

- 最小窗口下主流程仍可完成
- 关键按钮不能被挤出屏幕
- 表格列在小窗口下应支持横向滚动或隐藏次要列
- 日志区域需要可伸缩
- 配置预览区域需要可折叠或可调整宽度
- 1280 x 800 下应能完整呈现主导航、页面标题、核心操作区和主要内容
- 弹窗、抽屉、详情面板不能超过 1280 x 800 的可视范围
- 内容较多的页面应通过滚动、折叠、分栏收缩或详情面板解决，不依赖更大屏幕

## 16. MVP 范围边界

第一版需要设计，但不需要覆盖未来所有能力。

MVP 需要覆盖：

- Linux 节点管理
- SSH 连接
- FRP 自动下载
- FRP 核心上传
- TOML 配置生成
- TCP/UDP/HTTP/HTTPS 隧道
- 远程启动、停止、重启
- 日志查看
- 基础状态诊断

暂不作为第一版重点：

- Agent 模式
- 团队权限
- 云端同步
- Web Dashboard
- 计费系统
- Kubernetes
- 多租户
- 复杂流量分析
- 高级审计

未来能力可以在视觉结构上预留空间，但不要让它们干扰 MVP 主流程。
