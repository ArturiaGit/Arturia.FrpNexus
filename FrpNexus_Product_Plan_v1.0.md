# FrpNexus 产品方案 v1.0

## 1. 产品概述

FrpNexus 是一款面向个人开发者的开源 FRP 桌面部署与运维控制台。

它不是官网、营销页、云平台后台，也不是单纯的 FRP 配置编辑器。它的核心价值是把 FRP 的下载、部署、配置、启动、日志查看和基础诊断整合进一个桌面端，让用户不必频繁手写配置文件和 SSH 命令。

第一版产品重点支持：

- 通过 SSH/SFTP 管理远程 Linux 节点
- 自动下载或选择本地 FRP 核心文件
- 上传 `frpc` / `frps` 到远程节点
- 通过表单生成 TOML 配置
- 上传配置文件
- 启动、停止、重启远程 FRP 进程
- 查看日志、状态和基础错误信息

## 2. 产品定位

FrpNexus 定位为面向个人开发者的开源 FRP 桌面部署与运维控制台。

它要解决的问题不是“如何把 FRP 配置文件写得更漂亮”，而是“如何让个人开发者更轻松地完成 FRP 的远程部署、配置、运行和排障”。

产品不依赖云端账号，不以团队协作为第一目标，也不在第一版做复杂的商业化平台能力。它应该保持本地优先、开源透明、操作直接。

## 3. 产品价值

FrpNexus 的核心价值包括：

- 降低 FRP 上手成本
- 减少手写 TOML 配置的出错概率
- 减少重复 SSH 登录服务器的操作
- 统一管理多台远程 Linux 节点
- 统一管理 FRP 核心、配置、进程和日志
- 帮助用户更快定位连接失败、端口冲突、配置错误等问题

用户打开 FrpNexus 后，应该能够围绕一个目标完成闭环：把一个 FRP 服务或隧道真正跑起来，并能看到它是否正常。

## 4. 首要用户

第一版优先服务个人开发者。

典型用户包括：

- 有一台或多台 VPS 的开发者
- 想把本地 Web/API 服务临时暴露公网的开发者
- 想远程访问家中 NAS、SSH、RDP 或开发机的用户
- 自托管爱好者
- 熟悉基础网络概念但不想反复手写命令的人

第一版不优先服务大型企业运维团队、SaaS 平台运营者或需要复杂权限管理的组织。

## 5. 发布形态

FrpNexus 早期发布形态为开源桌面工具。

设计和产品表达上应强调：

- 本地运行
- 用户可控
- 开源透明
- 不强制绑定云端
- 不把商业能力放在主流程中

## 6. MVP 核心目标

MVP 的目标是让用户完成一次完整的远程 FRP 部署和运行管理。

MVP 必须让用户能够：

- 添加一台远程 Linux 节点
- 通过 SSH 测试连接
- 自动下载 FRP Release
- 根据节点系统和架构选择合适的 `frpc` / `frps`
- 通过 SFTP 上传 FRP 核心文件
- 通过表单生成 TOML 配置
- 上传配置文件
- 启动、停止、重启远程 FRP 进程
- 查看进程状态和远程日志
- 创建 TCP、UDP、HTTP、HTTPS 四类基础隧道

## 7. 核心模块

建议产品由以下模块组成：

- Dashboard：总览节点、隧道、进程和最近错误
- Nodes：管理远程 Linux 节点和 SSH 连接
- FRP Core：管理 FRP 版本下载、核心选择和上传
- Configurations：创建、预览、校验、上传 TOML 配置
- Tunnels：管理 TCP、UDP、HTTP、HTTPS 隧道
- Runtime：控制远程 FRP 进程启动、停止、重启
- Logs：查看和筛选远程日志
- Settings：管理下载源、默认路径、密钥、主题和本地数据

## 8. MVP 主流程

MVP 主流程如下：

1. 用户添加 Linux 节点。
2. 用户填写 Host、Port、User 和认证方式。
3. FrpNexus 测试 SSH 连接。
4. 用户选择 FRP 版本。
5. FrpNexus 下载 Release，并识别可上传的 `frpc` / `frps`。
6. 用户选择部署服务端或客户端。
7. FrpNexus 上传核心文件到远程目录。
8. 用户创建 `frps.toml` 或 `frpc.toml`。
9. FrpNexus 校验配置并展示 TOML 预览。
10. 用户上传配置。
11. 用户启动 FRP。
12. FrpNexus 显示运行状态和日志。
13. 用户根据状态继续重启、停止、修改配置或排查问题。

推荐默认远程目录：

```text
/opt/frpnexus/
  bin/
  config/
  logs/
  services/
```

## 9. 首批协议范围

MVP 首批支持以下代理类型：

- TCP
- UDP
- HTTP
- HTTPS

暂不把 STCP、SUDP、XTCP、TCPMUX、Visitor、插件模式作为第一版主流程。后续可以作为高级配置逐步加入。

## 10. 配置格式

FrpNexus 应以 TOML 作为第一版主要配置格式。

原因：

- FRP 新版本已支持 TOML、YAML、JSON
- INI 已属于废弃方向
- TOML 对表单生成、预览和结构化校验更友好

设计中应提供 TOML 预览面板，让用户知道表单最终会生成什么配置。

## 11. 远期路线

Phase 1：SSH/SFTP MVP

- Linux 节点管理
- SSH 连接测试
- FRP 自动下载
- 核心上传
- TOML 配置生成
- 远程启动、停止、重启
- 日志查看
- TCP/UDP/HTTP/HTTPS 隧道

Phase 2：桌面体验增强

- 系统托盘
- 深色/浅色主题
- 操作通知
- 配置导入/导出
- 常用模板
- 本地敏感信息加密存储

Phase 3：运维能力增强

- systemd 服务生成
- 开机自启管理
- 端口冲突检测
- FRP 配置验证
- 配置历史和回滚
- 多下载源或镜像源配置

Phase 4：Agent 模式

- FrpNexus.Agent
- Agent 心跳
- Agent 日志推送
- Agent 进程管理
- 一键部署 Agent
- 多节点实时监控

## 12. 第一版非目标

第一版不做：

- 团队权限
- 云端同步
- Web Dashboard
- 计费系统
- Kubernetes 集成
- 多租户
- 复杂流量分析
- 高级审计
- 完整 Agent 模式

这些能力可以在产品信息架构中预留扩展空间，但不能影响 MVP 主流程的简洁性。
